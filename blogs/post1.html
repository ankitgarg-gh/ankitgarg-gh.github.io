Designing a Data-as-a-Service Layer for Billion-Scale Events

December 2024

The Problem

When you're processing billions of events daily, the traditional approach of giving teams direct access to data lakes creates problems. Query costs spiral. Teams write inefficient queries. The same data gets scanned repeatedly by different consumers.

The alternative—building bespoke pipelines for each consumer—doesn't scale either. You end up with dozens of point-to-point integrations, each with its own maintenance burden.

Data-as-a-Service (DaaS) sits in the middle: a managed API layer that abstracts the underlying storage while providing controlled, efficient access patterns.

Architecture Overview

The system I'll describe serves approximately 5 billion events per day through a combination of REST and gRPC APIs. The underlying data lives in a combination of Hive tables on GCS and hot data in managed caches.

Key Design Decisions

1. REST vs gRPC: Use Both

The debate between REST and gRPC often misses the point. They serve different use cases:

  • REST: Better for ad-hoc queries, debugging, integration with external systems that expect HTTP/JSON
  • gRPC: Better for high-throughput streaming, internal service-to-service communication, when you control both ends

We expose both. The Query Service accepts REST requests for point lookups and bounded queries. The Stream Service uses gRPC for consumers that need continuous data feeds.

// Stream service definition
service EventStream {
  rpc Subscribe(SubscribeRequest) returns (stream Event);
  rpc GetCheckpoint(CheckpointRequest) returns (Checkpoint);
}

message SubscribeRequest {
  string topic = 1;
  string consumer_group = 2;
  Checkpoint start_from = 3;
}

2. Caching Strategy: Tiered by Access Pattern

Not all data is accessed equally. We use three tiers:

  • Hot (Redis): Last 24 hours, most-accessed entities. Sub-millisecond reads.
  • Warm (Materialized Views): Pre-aggregated data in BigQuery. Seconds for complex queries.
  • Cold (Hive/GCS): Full history. Minutes for large scans.

The Query Service routes requests to the appropriate tier based on the query pattern and freshness requirements.

public QueryResult executeQuery(Query query) {
    // Check if data is in hot cache
    if (isHotQuery(query)) {
        return redisClient.get(query.getCacheKey());
    }

    // Check if we have a materialized view
    MaterializedView view = viewRegistry.findView(query);
    if (view != null && view.isFresh(query.getFreshnessRequirement())) {
        return bigQueryClient.query(view.getQuery(query));
    }

    // Fall back to cold storage
    return hiveClient.query(query.toHiveQL());
}

3. Schema Versioning: Contract-First

With dozens of consumers, schema changes become dangerous. We use a contract-first approach:

  • Schemas are defined in Protocol Buffers
  • Each schema has a version number
  • The API supports requesting specific schema versions
  • Breaking changes require a new major version; old versions remain supported for a deprecation period

This decouples producers from consumers. A producer can evolve their schema without breaking existing consumers, as long as changes are backward-compatible.

Performance Results

After implementing this architecture:

  • Query costs reduced 40% — Caching and materialized views eliminated redundant scans
  • End-to-end latency reduced 30% — Hot path queries hit Redis instead of Hive
  • Consumer onboarding time reduced from weeks to days — Self-service API with documentation

What I'd Do Differently

Start with fewer tiers. We built three caching tiers from day one. In retrospect, starting with just hot and cold would have been simpler. We could have added the warm tier when we had clear evidence it was needed.

Invest more in observability earlier. Understanding query patterns is essential for cache tuning. We added comprehensive metrics late and had to make some caching decisions on intuition rather than data.

Consider a unified query language. Supporting both SQL-like queries and filter-based APIs created complexity. A single query interface (even if it compiles to different backends) would have simplified the client experience.
