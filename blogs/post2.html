Real-Time Pipelines in Regulated Industries

November 2024

Context

Building streaming systems is hard. Building them in regulated finance, healthcare, or government adds constraints that change your architecture fundamentally.

I spent five years building real-time data infrastructure in investment banking. The technical challenges were interesting, but the regulatory requirements shaped more decisions than any performance benchmark.

What Makes Regulated Different

1. Audit Everything

Every data access, transformation, and output must be traceable. Not just logged—provably traceable.

This means:

  • Immutable audit logs (append-only, tamper-evident)
  • Correlation IDs through entire pipeline
  • Retention policies that match regulatory windows (often 7+ years)
Event arrives   → Log receipt with timestamp, source, hash
Event processed → Log transformation applied, input hash, output hash
Event delivered → Log destination, delivery confirmation, timestamp

This isn't optional logging for debugging. It's evidence for regulators.

2. Exactly-Once Matters More

In ad-tech, processing an event twice means a slightly inflated count. In finance, processing a transaction twice means real money moves incorrectly.

Kafka's exactly-once semantics (EOS) are necessary but not sufficient. Your entire pipeline needs idempotency:

// Idempotent write pattern
public void processTransaction(Transaction txn) {
    String idempotencyKey = txn.getId() + "-" + txn.getVersion();

    if (processedTransactions.contains(idempotencyKey)) {
        log.info("Skipping duplicate: {}", idempotencyKey);
        return;
    }

    // Process within transaction
    try {
        db.beginTransaction();
        db.markProcessed(idempotencyKey);
        db.applyTransaction(txn);
        db.commit();
    } catch (Exception e) {
        db.rollback();
        throw e;
    }
}

3. Late Data Has Legal Implications

Stream processing frameworks have concepts of watermarks and late data handling. Usually, you configure a grace period—events arriving after the window closes are dropped or sent to a side output.

In regulated contexts, you often cannot drop late data. A trade that arrives late still happened. It needs to be recorded, reconciled, and potentially trigger corrections to previously published results.

This means:

  • Longer grace periods (higher latency)
  • Reprocessing capabilities (rebuild outputs from source)
  • Correction events (amendments, not overwrites)

4. Data Residency Is Real

GDPR, data localization laws, and internal policies mean you can't always send data where you want.

Architecture implications:

  • Regional Kafka clusters (not global replication)
  • Processing must happen in-region
  • Cross-region aggregations require anonymization or legal frameworks

Delta Lake was the right choice here. ACID transactions on the lake meant we could handle late arrivals with proper upserts. Time travel meant we could reproduce any historical state for audits. Schema enforcement meant bad data didn't corrupt downstream.

Latency vs Compliance Tradeoff

The business wanted sub-second latency. Compliance wanted bulletproof audit trails. These goals conflict.

The compromise:

  • Hot path: Sub-second for operational use (in-memory state, async persistence)
  • Warm path: Seconds for audit-ready writes (transactional commits to lake)
  • Reconciliation: Continuous comparison between hot and warm to catch drift

This added complexity but satisfied both stakeholders.

Key Lessons

  1. Start with compliance requirements, not performance benchmarks. The regulatory constraints are non-negotiable. Performance can be optimized within those bounds.
  2. Immutability is your friend. Append-only data models make audits trivial. You never have to explain what data looked like before an update because you never update in place.
  3. Build reprocessing capability from day one. You will need to rebuild outputs from source. If your pipeline can't replay history, you've built yourself into a corner.
  4. Invest in observability. When a regulator asks why a particular output was produced, you need to answer definitively. Distributed tracing and comprehensive logging aren't optional.
