Batch vs Stream: A Decision Framework

January 2025

The False Binary

Early in my career, I treated batch and stream as opposing philosophies. Batch was the old way—reliable, simple, slow. Stream was the future—modern, complex, fast.

After years of building these systems, I see them as complementary tools. The question isn't which paradigm is better. It's which constraints matter most for a specific problem.

The Framework

Four dimensions help decide:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│   Latency Requirement                                        │
│   ├── < 1 second     → Stream                               │
│   ├── < 1 hour       → Micro-batch or Stream                │
│   └── Daily/Weekly   → Batch                                │
│                                                              │
│   Completeness Requirement                                   │
│   ├── Must have all data before processing → Batch          │
│   └── Can handle late arrivals gracefully  → Stream         │
│                                                              │
│   Complexity of Transformation                               │
│   ├── Simple filters, maps, aggregations   → Either         │
│   ├── Complex joins across large datasets  → Batch          │
│   └── Windowed aggregations with state     → Stream         │
│                                                              │
│   Operational Maturity                                       │
│   ├── Team experienced with streaming      → Stream ok      │
│   └── Team new to streaming                → Start Batch    │
│                                                              │
└─────────────────────────────────────────────────────────────┘

Real Examples

1. Daily Business Metrics Dashboard

  • Latency: Users check it each morning. Daily is fine.
  • Completeness: Must include all of yesterday's data.
  • Complexity: Aggregations across multiple fact tables.
  • Team: Comfortable with Spark batch jobs.

Decision: Batch. No benefit to streaming here. A well-tuned overnight job is simpler and more reliable.

2. Fraud Detection

  • Latency: Must flag suspicious transactions in seconds.
  • Completeness: Can't wait for all data—need to act on each event.
  • Complexity: Pattern matching against recent history.
  • Team: Willing to invest in streaming expertise.

Decision: Stream. Latency requirement makes this non-negotiable.

3. Customer 360 Profile

  • Latency: Profiles should reflect recent activity within an hour.
  • Completeness: Needs to combine data from 12 source systems.
  • Complexity: Complex joins, deduplication, entity resolution.
  • Team: Mixed experience.

Decision: Lambda architecture. Batch for comprehensive daily rebuilds. Stream for real-time updates to hot entities. Serving layer merges both views.

Yes, Lambda is considered legacy by streaming purists. In practice, it works reliably for this use case with lower operational burden than pure streaming.

The Kappa Architecture Dream

The Kappa architecture—everything as a stream, batch as a special case of streaming—is theoretically elegant. Reprocess by replaying the stream. One codebase for both real-time and historical.

In practice, it works well for:

  • Log processing
  • Event sourcing systems
  • Simpler aggregation pipelines

It struggles with:

  • Complex multi-way joins
  • ML feature engineering with large historical windows
  • Teams without deep streaming expertise

The cost of getting Kappa wrong is high. Debugging stateful streaming applications is harder than debugging batch jobs. When a batch job fails, you rerun it. When a streaming job has incorrect state, you're in for archaeology.

Technology Choices

Current defaults:

Use Case Technology
Batch processing Spark on Kubernetes or managed (Dataproc, EMR, Databricks)
Stream processing Kafka Streams for simpler cases, Flink for complex stateful processing
Micro-batch Spark Structured Streaming
Storage Delta Lake or Iceberg (unified batch and stream sink)

Spark Structured Streaming deserves special mention. It's not pure streaming—it's micro-batch under the hood. But for many use cases where "real-time" actually means "within a few minutes," it provides streaming semantics with batch-like operational simplicity.

# Spark Structured Streaming example
df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "broker:9092") \
    .option("subscribe", "events") \
    .load()

processed = df \
    .selectExpr("CAST(value AS STRING) as json") \
    .select(from_json("json", schema).alias("data")) \
    .select("data.*") \
    .withWatermark("event_time", "10 minutes") \
    .groupBy(window("event_time", "5 minutes"), "category") \
    .agg(count("*").alias("event_count"))

processed.writeStream \
    .format("delta") \
    .outputMode("append") \
    .option("checkpointLocation", "/checkpoints/events") \
    .start("/delta/event_aggregates")

Key Takeaways

  1. Latency requirements are often negotiable. When stakeholders say "real-time," ask what they actually need. Often "within 15 minutes" satisfies the business need and dramatically simplifies the architecture.
  2. Operational cost matters more than theoretical elegance. A batch job that runs reliably for years beats a streaming job that needs constant babysitting.
  3. Start simpler, evolve when needed. Build batch first. If latency requirements genuinely can't be met, add streaming for that specific path. Don't stream everything because it's modern.
  4. Unified storage layers change the game. Delta Lake and Iceberg allow batch and stream to write to the same tables with ACID guarantees. This makes hybrid architectures much cleaner than they used to be.
  5. The best architecture is the one your team can operate. A theoretically superior design that your team struggles to debug and maintain is worse than a simpler design they understand deeply.
